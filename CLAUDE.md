## 0. Стандарты разработки 1С (v8std) — обязательный шаг перед разработкой

**Алгоритм работы со стандартами:**

1. **Читаю задачу** — определяю тип работы (запрос, форма, модуль, роли, обмен данными и т.д.)
2. **Определяю категорию** — нахожу релевантный файл по таблице ниже
3. **Читаю стандарты** — открываю `.md` файлы и читаю актуальный текст; не полагаюсь на обучающие данные
4. **Пишу код по стандартам** — реализую задачу с учётом прочитанных требований
5. **Проверяю перед коммитом** — сверяю написанный код с требованиями стандартов

**Соответствие типов задач разделам индекса:**

## 1. Именование

### Переменные и параметры

- Стиль: ВерблюжийРегистр (CamelCase) с заглавной буквы
- Имя отражает смысл, не тип: `НомерДокумента`, `ТекущийПользователь`
- Булевы переменные — с префикса: `ЭтоНовый`, `МожноРедактировать`, `ФлагПроведения`
- Префиксы `п`, `тз`, `вт`, `стр` и венгерская нотация — **ЗАПРЕЩЕНЫ**
- Буква `ё` в именах метаданных — **ЗАПРЕЩЕНА** (только в пользовательских сообщениях)

### Функции и процедуры

- Имя начинается с глагола: `ПолучитьСписокОборудования`, `УстановитьСтатус`, `ПроверитьПраваДоступа`
- Экспортные функции — обязательно с описанием (см. раздел Документирование)
- Максимум 5 параметров; если больше — использовать Структуру

### Объекты метаданных

- Справочники — во множественном числе: `Оборудование`, `ВидыРабот`
- Документы — в единственном числе: `НарядНаРаботу`, `ДефектнаяВедомость`
- Новые объекты — с префиксом проекта

### Временные таблицы в запросах

- Префикс `Вр`: `ВрОборудование`, `ВрВидыРабот`

## 3. Запросы

### Обязательные правила

- Запросы в циклах — **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНЫ**
- Точечная нотация для доступа к реквизитам в запросах — **ЗАПРЕЩЕНА** (только явные соединения)
- Подзапросы в SELECT — **ЗАПРЕЩЕНЫ**, использовать временные таблицы
- Фильтрация виртуальных таблиц — только через параметры виртуальной таблицы, не через ГДЕ
- Не проверять `.Пустой()` перед циклом по результату запроса — просто итерировать

### Оформление запросов

```bsl
Запрос = Новый Запрос;
Запрос.Текст =
    "ВЫБРАТЬ
    |   Оборудование.Ссылка КАК Ссылка,
    |   Оборудование.Наименование КАК Наименование
    |ИЗ
    |   Справочник.Оборудование КАК Оборудование
    |ГДЕ
    |   Оборудование.ВидОборудования = &ВидОборудования";

Запрос.УстановитьПараметр("ВидОборудования", ВидОборудования);
РезультатЗапроса = Запрос.Выполнить();
```

### Временные таблицы и пакетные запросы

- Для сложных запросов использовать менеджер временных таблиц
- Пакетные запросы предпочтительнее множества отдельных запросов
- Условие соединения (ПО) отделять от условия фильтрации (ГДЕ)

## 4. Транзакции и обработка ошибок

### Шаблон транзакции

```bsl
НачатьТранзакцию();
Попытка
    // код записи данных
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ЗаписьЖурналаРегистрации(
        НСтр("ru = 'ТОИР: Ошибка записи'"),
        УровеньЖурналаРегистрации.Ошибка,
        ,
        ,
        ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
    ВызватьИсключение;
КонецПопытки;
```

### Правила

- Пустые блоки `Исключение` — **ЗАПРЕЩЕНЫ**
- Для логирования использовать `ПодробноеПредставлениеОшибки()`, не `ОписаниеОшибки()`
- В подчинённых обработчиках не дублировать запись в журнал регистрации
- `НачатьТранзакцию()` — всегда ДО блока `Попытка`

## 5. Клиент-серверное взаимодействие

- Модальные вызовы — **ЗАПРЕЩЕНЫ**: `ВопросПользователю()`, `ОткрытьФормуСГарантированнымПолучениемРезультата()` и т.д.
- Использовать `&НаСервереБезКонтекста` вместо `&НаСервере` где возможно
- Минимизировать количество серверных вызовов из клиентского кода
- Не передавать на сервер объекты форм — только примитивные типы и структуры

---

## 6. Производительность и кэширование

- Повторно используемые данные — кэшировать через `РезультатыДляПовторногоИспользования`
- Получение реквизитов ссылки — через `ОбщегоНазначения.ЗначениеРеквизитаОбъекта()`, не через обращение к объекту
- Для множества реквизитов — `ОбщегоНазначения.ЗначенияРеквизитовОбъекта()`
- Функциональные опции читать через `ПолучитьФункциональнуюОпцию()`, не через константы напрямую

---

## 7. БСП (Библиотека стандартных подсистем)

- Использовать механизмы БСП вместо написания собственных аналогов
- Ключевые модули БСП для ТОИР:
  - `ОбщегоНазначения` — работа с реквизитами, типами, структурами
  - `ОбщегоНазначенияКлиент` — клиентские утилиты
  - `РаботаСФайлами` — файловые операции
  - `ПользователиКлиентСервер` — работа с пользователями
- Не переопределять и не дублировать функциональность БСП

## 8. Качество кода

### Структура и читаемость

- Максимальная длина метода: 200 строк
- Максимальная вложенность: 4 уровня
- Использовать "ранний возврат" (guard clauses) для сокращения вложенности
- Использовать `СтрШаблон()` для формирования строк вместо конкатенации
- Отступы — только TAB

### Комментарии

- Закомментированный код — **ЗАПРЕЩЁН** (для этого есть git)
- Комментарии объясняют **ПОЧЕМУ**, а не ЧТО
- Обязательны для нетривиальной бизнес-логики

### Антипаттерны (ЗАПРЕЩЕНЫ)

- Запрос в цикле
- Точечная нотация в запросах
- Подзапросы в SELECT
- Фильтр виртуальной таблицы через ГДЕ
- Избыточные серверные вызовы
- Повторное использование переменных с разным смыслом
- God-модуль (модуль, который делает всё)
- Copy-Paste архитектура

## 12. Стандарты разработки 1С (справочник)

Справочник стандартов разработки 1С от Ингвара — [github.com/zeegin/v8std](https://github.com/zeegin/v8std).

Стандарты разбиты по разделам и хранятся в `standarts/std/`. Перед началом работы над задачей — читай соответствующий файл.

### Карта разделов


| Задача                                  | Файл стандартов                 |
| --------------------------------------- | ------------------------------- |
| Пишу запрос, работаю с данными          | `standarts/std/queries.md`      |
| Разрабатываю форму, элементы управления | `standarts/std/forms.md`        |
| Строю командный интерфейс, меню, панели | `standarts/std/ui.md`           |
| Пишу код на встроенном языке            | `standarts/std/language.md`     |
| Добавляю обработчики событий            | `standarts/std/events.md`       |
| Создаю или меняю объекты метаданных     | `standarts/std/metadata.md`     |
| Оптимизирую, работаю с блокировками     | `standarts/std/performance.md`  |
| Настраиваю роли и права доступа         | `standarts/std/security.md`     |
| Разрабатываю отчёт или печатную форму   | `standarts/std/reports.md`      |
| Создаю регламентное задание             | `standarts/std/scheduled.md`    |
| Оформляю интерфейс, тексты, стили       | `standarts/std/ux.md`           |
| Добавляю локализацию                    | `standarts/std/localization.md` |
| Настраиваю интеграцию, обмен данными    | `standarts/std/integration.md`  |
| Работаю с исключениями и журналом       | `standarts/std/logging.md`      |


### Правило применения

1. Определи категорию задачи по таблице выше
2. Прочитай соответствующий файл из `standarts/std/`
3. Если стандарт противоречит правилам этого файла — приоритет у этого файла
4. Если стандарт дополняет — применяй оба
